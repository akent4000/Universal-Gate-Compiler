## 🔴 Критические ограничения

### 1. Экспоненциальная сложность QMC

`quine_mccluskey` хранит все импликанты в памяти и итерирует попарно. При `n > 16` это взрывается.

```python
# implicant.py — текущий подход
while current:
    for i in range(len(current)):
        for j in range(i + 1, len(current)):   # O(|current|²) на каждой итерации
            if current[i].can_combine(current[j]):
```

**Что сделать:** группировать импликанты по числу единиц в битовом векторе — тогда нужно сравнивать только соседние группы (`popcount(a) + 1 == popcount(b)`), что даёт на практике 10-100× ускорение:

```python
# Группировка по весу Хэмминга
from collections import defaultdict
groups = defaultdict(list)
for imp in current:
    groups[imp.bits.count(1)].append(imp)

for w in sorted(groups):
    for a in groups[w]:
        for b in groups[w + 1]:   # только соседние группы
            if a.can_combine(b): ...
```

### 2. Жадное покрытие не оптимально

`select_cover` после существенных импликант делает жадный выбор — это не гарантирует минимальное покрытие:

```python
# optimize.py — можно пропустить глобально лучшее решение
best = max(
    (pi for pi in primes if pi not in selected),
    key=lambda pi: (len(pi.covered & remaining), -pi.literal_count()),
)
```

Для задач до ~20 простых импликант и ~20 минтермов точное решение даёт **метод Петрика** (Petrick's method) или LP-релаксация. Жадность может проигрывать 1–3 гейта на реальных схемах.

### 3. Фазовое назначение независимо по выходам

```python
# pipeline.py
for idx, name in enumerate(tt.output_names):
    result.outputs[name] = _optimize_output(tt, idx, builder, verbose)
```

Каждый выход выбирает полярность (`f` или `~f`) изолированно. Но NAND-builder глобальный — если выход A уже построил `NAND(x, y)`, то выход B, которому нужен `x & y`, получит его бесплатно. Текущий код этого не учитывает при принятии решения о полярности.

## 🟡 Алгоритмические улучшения

### 4. Факторизация работает только на одном литерале

`factorize_once` ищет **один литерал**, встречающийся в нескольких термах. Это первый уровень алгебраической факторизации. Полноценная факторизация по **ядрам и со-ядрам** (Brayton, 1982) извлекает общие *кубы*, а не просто литералы:

```
Текущий результат:  x·a·b | x·a·c  →  x·a·(b | c)   ✓
Пропускается:       a·b·c | a·b·d | e·f  →  a·b·(c | d) | e·f
```

### 5. Декомпозиция Шеннона — только один уровень

```python
def apply_shannon(expr, var_names):
    best = expr
    for v in var_names:
        candidate = shannon_on_var(expr, v)  # один проход по всем переменным
        if candidate.literals() < best.literals():
            best = candidate
    return best
```

Нет рекурсии и нет перебора пар переменных. Двухуровневая декомпозиция (`shannon_on_var` рекурсивно на кофакторах) часто даёт ещё 10–20% сокращение.

### 6. Нет детектирования XOR/XNOR

`a ⊕ b` в NAND-базисе требует 4 гейта. Если в выражении появляется паттерн `(a & ~b) | (~a & b)`, его стоит распознавать явно — иначе структурный хэшинг не поможет, потому что буквальные деревья для разных входов будут разными:

```python
def detect_xor(expr: Expr) -> Expr:
    """Распознать (a & ~b) | (~a & b) → XOR(a, b) как атомарный узел."""
    ...
```

### 7. Булева реsubstitution отсутствует

После построения NAND-сети можно проверить: не является ли некий промежуточный wire `wN` логически эквивалентным другому или его дополнению на подпространстве don't-care. Это позволяет заменить целые поддеревья уже посчитанными сигналами.

## 🔵 Производительность кода

### 8. `eval_network` — словарь на каждый гейт

```python
def eval_network(gates, inputs):
    wires: Dict[str, int] = dict(inputs)
    for name, gtype, ins in gates:
        if gtype == 'NAND':
            wires[name] = 1 - int(all(wires.get(i, 0) == 1 for i in ins))
```

При запуске тестов это вызывается `2^n × n_outputs` раз. Для 7-seg: `16 × 7 = 112` вызовов, каждый — словарный поиск по строкам. Простое ускорение — скомпилировать сеть в список лямбд или numpy-массив один раз:

```python
def compile_network(gates, input_names):
    """Вернуть функцию (input_vector) → output_vector, скомпилированную один раз."""
    ...
```

### 9. `factorize_once` — O(terms × literals²)

```python
lit_occ: Dict[Tuple[str, bool], List[int]] = {}
for i, ts in enumerate(term_sets):
    for lit in ts:           # для каждого литерала в каждом терме
        lit_occ.setdefault(lit, []).append(i)
```

При большом числе термов это пересчитывается каждый раз в цикле `factorize(..., rounds=10)`. Можно строить `lit_occ` инкрементально после каждой факторизации.

### 10. `select_cover` — пересчёт `coverage` при каждом вызове

```python
def select_cover(primes, ones):
    coverage: Dict[int, List[Implicant]] = {m: [] for m in ones}
    for pi in primes:
        for m in ones:
            if m in pi.covered: ...
```

`pi.covered` — `frozenset`, проверка `m in pi.covered` — O(1), но внешний цикл `primes × ones` вызывается каждый раз при вызове `optimize`. Можно кэшировать coverage как часть `Implicant`.

## 🟢 Архитектурные предложения

### 11. Многовыходная факторизация

Сейчас `factorize` работает на одном выражении. Если передать все выходы сразу, можно извлекать **общие подвыражения между выходами** на уровне Expr-деревьев — до того как NAND-builder получит их по отдельности:

```python
def multi_output_factorize(exprs: Dict[str, Expr]) -> Dict[str, Expr]:
    """Найти подвыражения, встречающиеся в ≥2 выходах, вынести в shared узлы."""
    ...
```

### 12. Верификация через SAT вместо полного перебора

Тест T9 проверяет все `2^n` входных комбинаций. Для `n > 20` это невозможно. Правильный подход — `miter circuit` + SAT-solver: построить схему `original ⊕ optimized` и доказать, что она тождественно 0. Python-библиотека `pysat` позволяет это сделать без внешних зависимостей.

### 13. Порядок этапов pipeline

Текущий порядок: `QMC → Phase → Factor → Shannon → InvElim → NAND`

Более эффективный порядок с обратной связью:

```
QMC → Phase → Factor → Shannon → Factor(again) → InvElim → NAND
                ↑__________________________|
```

После Shannon часто появляются новые возможности для факторизации — один дополнительный вызов `factorize` после него может дать ещё 5–15% сокращение литералов.
