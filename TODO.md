
#  Дорожная карта развития Universal-Gate-Compiler (EDA Synthesizer)

Этот документ описывает эволюцию компилятора логики от базового оптимизатора до масштабируемого промышленного инструмента логического синтеза. Задачи сгруппированы по логическим фазам развития.

## Фаза 1: Инфраструктура, Бенчмарки и Надежность
*Надежный фундамент критически важен, так как сложные оптимизации на графах (AIG) очень легко ломают логику схемы.*

* [x] **Расширение базы MCNC-бенчмарков:** * Внедрить индустриальные стресс-тесты для отслеживания регрессий качества: `RD53` (popcount), `Parity` (9-bit XOR-деревья), умножители `3x3` и `4x4`, а также плотные автоматы `misex1` и `z4ml`. 
  * Эти тесты покажут реальную эффективность синтезатора на различных типах логики (арифметика vs. управление).
* [x] **Формальная верификация (Miter-схемы через SAT):** * Полный перебор (Exhaustive simulation) не работает для схем $>20$ входов. Необходимо написать функцию формальной проверки эквивалентности через `z3-solver`. 
  * **Как это работает:** Строится "miter" (схема-сравнение), где выходы исходной таблицы истинности и сгенерированной NAND-сети попарно соединяются через `XOR`, а их результаты объединяются через `OR`. Если SAT-солвер выдает `UNSAT` (невозможно получить 1 на выходе), схемы гарантированно идентичны.
* [x] **Property-Based Testing (Случайные графы):** * Использовать библиотеки (например, `Hypothesis` для Python) для генерации случайных мутаций таблиц истинности и фрагментов AIG с автоматической проверкой их эквивалентности после агрессивных оптимизаций.
* [x] **Профилирование производительности (Profiling):** * Внедрить замеры времени исполнения и потребления памяти для каждого прохода (Espresso, Factorization, Rewriting). Это поможет выявлять экспоненциальные взрывы алгоритмов на ранних этапах.

## Фаза 2: Булевский и Структурный Синтез (Текущий этап)
*Текущий конвейер требует усиления математической базы для преодоления порога в ~300 гейтов на сложных бенчмарках (напр. BCD-to-7seg).*

* [x] **Многовыходовая минимизация (Multi-output Espresso):** Общие кубы успешно извлекаются.
* [x] **Извлечение ядер и ко-ядер (Kernels and Co-kernels):** Реализован классический алгоритм факторизации Бреятона (`brayton_factor` в `optimize.py`) — перечисление ядер по Brayton-McMullen, алгебраическое деление, выбор лучшего ядра по выигрышу литералов и рекурсивная факторизация. Пример `a*c + a*d + b*c + b*d` сжимается в `(a + b) * (c + d)` (8 → 4 литералов).
* [x] **Функциональная декомпозиция (Ashenhurst-Curtis / Roth-Karp):** Реализован алгоритм поиска независимых подфункций $f(X) = g(h_1(X_{bound}), \ldots, h_k(X_{bound}), X_{free})$ в [nand_optimizer/decomposition.py](nand_optimizer/decomposition.py) — перебор бипартиций входов, тест множественности столбцов декомпозиционной диаграммы Рота-Карпа ($\mu \le 2^k$), жадная кластеризация строк с don't-cares, синтез $h_i$ и $g$ через Espresso, выбор по литерному выигрышу. На RD53-popcount даёт 80 → 24 литерала для LSB.
  * **Цель:** Позволить алгоритму автоматически находить логические "бутылочные горлышки" внутри плоских таблиц истинности (например, выявить, что схема вычисляет промежуточный 8-битный код перед финальным ответом) и разбивать задачу на две независимые, радикально уменьшая общую площадь.
* [x] **Exact Synthesis (SAT-based Boolean Matching):** Реализован SAT-солвер (Z3) в [nand_optimizer/exact_synthesis.py](nand_optimizer/exact_synthesis.py) — параллельный per-minterm Boolean-энкодинг с one-hot селекторами портов и симметрийным правилом `l_idx < r_idx`, итеративный поиск по возрастанию числа AND-гейтов. Результаты кешируются в памяти и сразу используются рерайтером (`rewrite_aig(..., use_exact=True)`). На проверках: XOR₂ → 3 AND, MAJ₃ → 4 AND, 3-XOR → 6 AND, все с верификацией через `evaluate_template`.

## Фаза 2.5: Технический долг и исправление маппинга (Hotfixes)
*Критический пул задач по устранению багов, приводящих к неоправданному раздуванию NAND-сетей (откат качества на этапе AIG).*
* [x] **Учет Global Fanout в AIG Rewriter (Причина раздувания №1):** Полностью переписан [nand_optimizer/rewrite.py](nand_optimizer/rewrite.py) с эвристикой, осознающей fanout. Подсчитываются ссылки (`_compute_ref_counts`) по всему AIG включая выходы; для каждого кандидата-cut вычисляется MFFC (Maximum Fanout-Free Cone) через стандартный kill-ref алгоритм, а стоимость нового шаблона считается non-destructive через виртуальные литералы (`_count_template_new_nodes`) с учётом intra-template sharing и структурного хеша нового AIG. Замена допускается только при строгом `n_new < mffc_size`. На бенчмарках без регрессий и с реальными выигрышами: misex1 75→58 (−23%), rd53 51→47 (−4), adder 20→19; 7seg/excess3/z4ml/parity9/mult3 без изменений; miter-верификация подтверждает эквивалентность. Пайплайн активирован (`rounds=1`).
* [x] **Продвинутый AIG-to-NAND маппинг (Bubble Pushing / Причина №2):** Реализован pre-pass `_bubble_push` в `aig_to_gates` ([nand_optimizer/nand.py](nand_optimizer/nand.py)). Алгоритм ищет паттерн `AND(pos_v, c)` где `pos_v` — позитивный полярный вход (требует инвертора). Если в AIG уже есть узел `m = AND(a, c)` или `AND(b, c)` (где `a,b` — входы `v`) с `pos_m` уже необходимым другому потребителю — реструктурирует узел на `AND(b, pos_m)`, устраняя отдельный инвертор для `pos_v`. Итеративный до фиксированной точки (≤8 раундов), с топологической проверкой безопасности. Результат на Binary_to_7seg: **381 → 373 гейтов** (−8), инверторы **105 → 100** (−5). Все тесты, верификация z3 и бенчмарки MCNC проходят без регрессий.
* [x] **Очистка фантомных узлов (AIG Garbage Collection):** Реализован метод `AIG.gc(out_lits)` ([nand_optimizer/aig.py](nand_optimizer/aig.py)): DFS от выходных литералов помечает достижимые узлы, затем перестраивает компактный AIG только из них — обе структуры `_nodes` и `_hash` содержат исключительно живые узлы. `rewrite_aig` вызывает `gc` в конце каждого раунда ([nand_optimizer/rewrite.py](nand_optimizer/rewrite.py)), устраняя мёртвые трансляции MFFC-потомков. Побочный эффект: структурный хеш в следующем раунде не засорён фантомными записями, поэтому cost-model `_count_template_new_nodes` точнее оценивает коллизии.
* [x] **Прямая генерация AIG (Устранение AST-оверхеда):** * Убрать промежуточные конвертации логики: `AST -> Legacy Builder -> AIG`. Строить AIG сразу после минимизации (или парсинга). Логические оптимизации (например, разложение Шеннона) выполнять исключительно как быстрые перекидывания указателей на графе.
* [x] **Извлечение XOR/XNOR-структур (Boolean Division):** * Алгоритмы типа Espresso сплющивают всю логику в сумму произведений (SOP). Для функций чётности (XOR) это приводит к экспоненциальному взрыву числа литералов. 
  * **Решение:** Реализован проход `_detect_xor_patterns` в [nand_optimizer/nand.py](nand_optimizer/nand.py). Сканирует все AND-узлы в AIG на наличие паттерна `n3=AND(~n1,~n2)` где `n1=AND(p,q)`, `n2=AND(p^1,q^1)`. Проверяет эксклюзивность: `n1` и `n2` используются только `n3` (fanout=1, positive polarity не нужна). Для найденных паттернов в `aig_to_gates` вместо стандартной 5-NAND формы (с `NOT(x)` и `NOT(y)`) эмитируется канонический 4-NAND XOR: `g1=NAND(p,q^1)`, `g2=NAND(p,g1)`, `g3=NAND(q^1,g1)`, `out=NAND(g2,g3)`. На parity9 извлекается 4 XOR-паттерна; верификация через z3 подтверждает эквивалентность на всех MCNC-бенчмарках.

## Фаза 2.7: Продвинутые манипуляции с графами (Advanced AIG)
*Инструментарий промышленного качества для работы с AIG.*

* [x] **FRAIGing (Functionally Reduced AIGs):** Реализован в [nand_optimizer/fraig.py](nand_optimizer/fraig.py) (Mishchenko et al., 2005). Пайплайн: (1) bit-parallel псевдослучайная симуляция всех узлов AIG с W-битной сигнатурой, (2) разбиение узлов на классы по сигнатуре (включая классы «равен константе»), (3) для каждой пары-кандидата Z3-miter доказывает эквивалентность (UNSAT ⇒ merge), при SAT — контрпример добавляется в паттерны и классы пересчитываются, (4) реконструкция компактного AIG с заменой слитых узлов и `gc`. Интегрирован в [pipeline.py](nand_optimizer/pipeline.py) между rewrite и balance; убирает скрытую избыточность, недоступную алгебраическим проходам.
* [x] **AIG Balancing (Балансировка графа):** * Написать алгоритм реструктуризации графа для минимизации его логической глубины (сокращение числа последовательных узлов) с сохранением площади. Критически важно для оптимизации задержек распространения сигнала (Timing). Реализован в [nand_optimizer/balance.py](nand_optimizer/balance.py): сбор листьев AND-дерева с поглощением однофанаутных узлов + min-heap для построения минимально-глубокого дерева. Площадь сохраняется: n листьев → n−1 AND. Интегрирован как проход [8] в `pipeline.py`, параметр `balance=True`.
* [x] **Скрипты синтеза (Synthesis Scripts):** Реализован ABC-подобный парсер и исполнитель в [nand_optimizer/script.py](nand_optimizer/script.py). Поддерживаемые команды: `balance`, `rewrite [-z] [-r N] [-K N]`, `refactor` (alias rewrite), `fraig`; флаги управляют числом раундов, размером cut и включением exact synthesis (`-z`). Скрипт передаётся в пайплайн параметром `script=...` (заменяет фиксированную цепочку rewrite→fraig→balance); default — `"rewrite; fraig; balance"`. Позволяет динамически подбирать цепочки под конкретную схему, напр. `balance; rewrite; refactor; balance; rewrite -z`.
* [x] **Визуализация (Graphviz Export):** * Добавить функцию дампа структуры AIG в формат `.dot`. Визуальный анализ узких мест схемы бесценен для отладки компилятора.
* [x] **Precomputed 4-input AIG database (NPN-lookup для рерайтера):** В [nand_optimizer/precompute_4cut.py](nand_optimizer/precompute_4cut.py) реализован генератор оптимальных AIG-шаблонов для всех $2^{16}$ булевых функций от ≤4 переменных: BFS по возрастанию числа AND-гейтов с мемоизацией по TT (16-битным векторам на литералах $x_0, x_1, x_2, x_3$), пулы по стоимости и восстановление структуры через `lit_to_args`. Результат сериализован в [nand_optimizer/aig_db_4.py](nand_optimizer/aig_db_4.py) как `AIG_DB_4: Dict[TT, template]` и используется рерайтером ([rewrite.py:358-359](nand_optimizer/rewrite.py#L358-L359)) как быстрый lookup для cut'ов размера ≤4 — до обращения к дорогому SAT-based exact synthesis. Для cut'ов $k=5,6$ экзакт-синтез кеширует результаты в памяти и подключается по требованию.
* [ ] **Don't-Care оптимизация (ODC/SDC):** * Вычисление observability don't-cares (ODC — паттернов входов, при которых выход узла не влияет на PO) и satisfiability don't-cares (SDC — невозможных комбинаций на внутренних сигналах). Используется для локального упрощения узлов сверх структурных методов: в окне узла минимизируется функция, расширенная don't-care-множеством. Ссылка: Mishchenko et al., "Scalable Don't-Care-Based Logic Optimization and Resynthesis" (2009). Встраивается между FRAIG и rewrite, даёт дополнительный ~5-15% площади на плотной логике.
* [ ] **XAG (XOR-AND Graph):** * Ввести XOR как первоклассный узел графа (в дополнение к AND с инверсиями). Сейчас XOR-паттерны детектируются только в [nand.py](nand_optimizer/nand.py) на этапе маппинга — это поздно: rewrite/FRAIG видят XOR как 3 AND и не могут нормально оптимизировать арифметику/криптографию. XAG естественно представляет сумматоры, компараторы, parity-функции и SHA-подобные схемы. Потребует адаптации AIG-API и базы NPN-классов (XAG_DB).
* [ ] **MIG (Majority-Inverter Graph):** * Альтернативное представление через 3-входовый мажоритарный элемент MAJ(a,b,c) и инверсии. Аксиоматика MIG (commutativity, associativity, distributivity, inverter propagation) даёт более компактное представление сумматоров и умножителей, чем AIG. Аспирационно — полезно как экспериментальный бэкенд для арифметических бенчмарков. Ссылка: Amarú et al., "Majority-Inverter Graph: A New Paradigm for Logic Optimization" (TCAD 2016).
* [ ] **SAT sweeping и window-based оптимизация:** * Более глубокая версия FRAIG: выделяется ограниченное окно (cone глубины ≤ D вокруг узла), для него вычисляются локальные don't-cares, затем SAT-solver минимизирует функцию окна и реструктуризирует внутренние узлы. Позволяет убирать избыточность, не обнаруживаемую глобальной сигнатурной симуляцией FRAIG.

## Фаза 3: Синтез последовательностной логики (FSM / RTL)
*Превращение проекта из простого комбинаторного оптимизатора в полноценный компилятор конечных автоматов и регистровых передач.*

* [ ] **Структура `StateTable` (STT):** * Создать класс для описания автоматов Мура/Мили: текущие состояния ($Q_n$) + входы $\rightarrow$ следующие состояния ($D_{n+1}$) + выходы.
* [ ] **Примитивы памяти:** * Добавить поддержку узлов `D_FF` (D-триггер) и `JK_FF` в `NANDBuilder`. Интегрировать их поддержку в генератор `.circ` файлов для Logisim.
* [ ] **Алгоритмы кодирования состояний (State Encoding):** * Автоматическое назначение битовых векторов абстрактным состояниям автомата. Поддержка стратегий: Binary (плотная упаковка), One-Hot (для быстрых автоматов на FPGA), Gray code (для минимизации переключений).
* [ ] **Генерация функций возбуждения (Excitation Logic):** * Вычисление логики для входов триггеров ($D$ или $J, K$) на основе выбранной кодировки и таблицы переходов.
* [ ] **Разрыв комбинаторных циклов (Feedback Loop Handling):** * Топологический сортировщик (симулятор) должен уметь корректно обрабатывать обратные связи через элементы памяти, чтобы не уходить в бесконечный цикл.
* [ ] **Минимизация состояний (Hopcroft partition refinement):** * Классический $O(n \log n)$ алгоритм разбиения состояний на классы эквивалентности: стартовое разбиение по выходной реакции, затем итеративное дробление по тому, в какие классы ведут переходы на одинаковых входах. Выполняется *до* кодирования — убирает избыточные состояния, которые пользователь объявил раздельными, но которые ведут себя идентично. Обязательный первый проход после парсинга STT.
* [ ] **Неполностью специфицированные автоматы (Incompletely-Specified FSM):** * Поддержка don't-care переходов и don't-care выходов в STT. Минимизация состояний при DC сводится к задаче покрытия (NP-hard), но существуют эффективные эвристики (STAMINA-подобные). Открывает возможность синтеза FSM из частичных спецификаций.

## Фаза 4: Масштабирование алгоритмов (High-Performance EDA)
*Для работы с функциями от 20+ переменных ($N \ge 20$), где массивы таблиц истинности начинают потреблять гигабайты RAM.*

* [x] **Исчисление кубов (Cube Calculus):** * **КРИТИЧНО:** Полностью отказаться от генерации и хранения всех $2^N$ минтермов в `truth_table.py` и `implicant.py`. Переписать всё на работу с массивами кубов (строки вида `0, 1, -`). Это устранит экспоненциальную сложность по памяти.
* [ ] **Интеграция ROBDD / BDD:** * Использовать библиотеки Binary Decision Diagrams (например, пакет `dd`) в качестве внутреннего ядра для компактного представления логики. BDD позволяет мгновенно проверять эквивалентность и эффективно вычислять кофакторы Шеннона.
* [ ] **Анализ критического пути (Static Timing Analysis - STA):** * Вычисление максимальной логической задержки (Arrival Time) от входов до выходов. Построение графа slack'ов (временны́х запасов) для фокусной оптимизации самых длинных путей.
* [ ] **Delay/Area Trade-off (Оптимизация с ограничениями):** * Добавить эвристики, которые позволяют пользователю выбирать: сгенерировать глубокую узкую цепочку (максимальная экономия гейтов) или широкое сбалансированное дерево (максимальное быстродействие за счет большего числа вентилей).
* [ ] **Параллелизация проходов (Multi-process rewrite/FRAIG):** * Rewrite и FRAIG тривиально распараллеливаются по независимым cone'ам выходов: для каждого cone'а — своя копия состояния, merge через общую базу структурного хеша. На больших AIG (>10k узлов) даёт почти линейный speedup без изменения качества результата. Использовать `multiprocessing` + разделяемый менеджер для db.
* [ ] **Incremental synthesis (инкрементальные проходы):** * Отслеживать "грязные" подграфы между итерациями скрипта синтеза (`balance; rewrite; fraig; balance; rewrite`) и перезапускать проход только на изменённой части AIG. Резко сокращает стоимость итеративной отладки скриптов и длинных цепочек оптимизации.

## Фаза 5: Входные языки и Технологический маппинг
*Интеграция инструмента во внешний мир электроники.*

* [ ] **Verilog Front-end:** * Написать парсер для подмножества структурного и простого поведенческого Verilog (с использованием `pyverilog` или аналогов). Это позволит компилировать реальные аппаратные модули, а не только абстрактные `.pla` файлы.
* [ ] **Standard Cell Mapping (Библиотечный маппинг):** * Отойти от жесткой привязки к `NAND`. Реализовать алгоритм покрытия дерева (Tree Mapping / Dynamic Programming) для трансляции AIG в произвольную библиотеку логических элементов (AND, OR, NOR, AOI, OAI, MUX).
* [ ] **LUT-маппинг (FPGA Technology Mapping):** * Группировка цепей AIG в K-входовые макроячейки (Look-Up Tables) для прямого синтеза под архитектуры FPGA (например, 4-LUT или 6-LUT, применяемые в Xilinx/Altera).
* [ ] **Имитация отжига (Simulated Annealing):** * Внедрить вероятностные алгоритмы переподключения узлов во время маппинга для выхода из локальных минимумов оптимизации.
* [ ] **AIGER / BLIF I/O (индустриальные форматы обмена):** * Поддержка чтения/записи AIGER (бинарный формат AIG, используемый в ABC, aiger-toolkit, EPFL) и BLIF (Berkeley Logic Interchange Format, стандартный текстовый формат для сетей SOP/узлов). Откроет прямой интероп: можно гонять свои AIG через ABC для cross-validation, импортировать результаты Yosys, сравниваться на публичных контестах.
* [ ] **EPFL Combinational Benchmark Suite:** * Подключить современный де-факто-стандарт для оценки синтезаторов: EPFL benchmarks делятся на arithmetic (adder, multiplier, square-root, divisor, sin) и random/control (priority encoder, router, voter, memory controller). Позволит напрямую сравнивать качество с ABC, Mockturtle и публикуемыми результатами. Требует `AIGER` I/O (см. пункт выше).

## Фаза 6: Тестопригодность, мощность и физическая реализация (Testability & Physical)
*Слой, который отделяет учебный оптимизатор от инструмента, пригодного для реального ASIC-флоу.*

* [ ] **ATPG (Automatic Test Pattern Generation):** * Генерация тестовых векторов для модели stuck-at-faults (константное "залипание" сигнала на 0 или 1). Реализация через SAT-кодирование miter'а "faulty vs. fault-free": каждая цель-неисправность даёт SAT-задачу поиска входа, различающего две версии схемы. Покрытие (fault coverage) — обязательная метрика для ASIC-тейпаута. Альтернатива: классический D-алгоритм / PODEM для структурного ATPG.
* [ ] **Оценка переключательной активности (Switching Activity Estimation):** * Распространение вероятностей переключений $p(sw)$ по AIG от входов (с заданным распределением/статистикой паттернов) ко всем внутренним узлам через правила для AND: $p_{sw}(y) = p(y)(1-p(y))$ с учётом (не)коррелированности входов. Основа для power-aware синтеза: высокоактивные узлы — кандидаты на минимизацию глитчей, buffer-insertion, перестройку дерева.
* [ ] **Power-aware optimization:** * Использовать оценки активности для выбора вариантов рерайтинга: при равной площади предпочитать структуры с меньшим switching × capacitance на узлах. Также — glitch reduction через балансировку путей (сигналы с разными arrival time на AND-входах генерируют импульсы).
* [ ] **Design-for-Test (DFT) — scan-chain insertion:** * После реализации Фазы 3 (D-триггеры) — автоматическая вставка scan-chain: замена обычных D-FF на Scan-FF с мультиплексором на входе (test/functional mode), соединение их в последовательную цепочку. Scan-chain делает все состояния автомата наблюдаемыми и управляемыми, без чего ATPG на последовательной логике нереалистичен.
* [ ] **Анализ тестопригодности (Testability metrics — SCOAP):** * Вычисление controllability (сложности установить узел в 0/1 со входов) и observability (сложности наблюдать узел на выходах) по формулам Sandia Controllability/Observability Analysis Program. Метрики помогают находить "слепые зоны" схемы и подсказывают, куда ставить test points.
